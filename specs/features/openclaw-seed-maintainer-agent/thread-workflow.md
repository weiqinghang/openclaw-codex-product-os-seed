# 线程工作流

## 元数据

- 功能：飞书直连的 seed 项目维护 Agent
- 场景：飞书线程中的仓库维护请求

## 1. 主流程

1. 人类发起 seed 仓库维护请求
2. `龟丞相` 判断是否属于本仓库范围
3. `龟丞相` 按请求类型加载最少必要工件
4. `龟丞相` 输出下一步建议或整理后的执行项
5. 需要实际改动时，再准备交给 `Codex` 的任务包
6. 更新仓库工件后，向人类汇报结果和阻塞点

## 2. 首条消息处理

当飞书消息较短或模糊时，先做两件事：

1. 判断它属于哪个工作类别
2. 判断它是继续已有迭代，还是开启新切片

禁止直接把整仓库文档整包注入上下文。

## 3. 消息分类到动作

### 规划类

- 读取 `story map` 和 `current iteration`
- 输出建议主题、范围和关联工件

### 实施类

- 读取当前 feature 工件
- 收敛为最小实施批次
- 准备可交给 `Codex` 的执行任务

### 验收类

- 读取验收标准、当前状态和已改工件
- 输出差距、风险和下一步验证动作

### 用法或治理类

- 读取对应 docs 分区
- 只更新相关分区，不扩大范围

## 4. 默认推进边界

默认只做这些事：

- 收敛任务
- 更新工件
- 准备变更
- 汇报状态

默认不做这些事：

- 自动提交
- 自动 push
- 扩大到其他仓库

## 5. 汇报格式

每次线程汇报用固定四段：

1. 当前状态
2. 已处理工件
3. 下一步建议
4. 待人类确认

## 6. 升级条件

以下情况升级到人类：

- 要改本机 `OpenClaw` 配置
- 要改变 Agent 行为边界
- 要进入仓库外系统
- 要把模式推广到其他仓库

以下情况升级到 `Codex`：

- 需要真实文件修改
- 需要结构化重组
- 需要批量落仓
- 需要验证修改结果
