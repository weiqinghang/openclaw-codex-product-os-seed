## 1. 问题

当前 seed 仓库已经形成主线文档、能力包和 backlog，但还没有一个专门的 `OpenClaw` 入口来持续维护它。人类若想继续推进仓库演进，仍需要手工解释上下文、临时决定加载哪些文档、以及临时组织对 `Codex` 的 `ACP` 调用。

## 2. 用户与场景

用户是维护本仓库的人类。场景是人类在飞书中直接提出“继续完善 seed 项目”的请求，希望 `OpenClaw` 能识别这是 seed 维护工作，而不是普通聊天或其他业务项目。

## 3. 期望结果

交付后应至少达到：

- 存在一个专门面向 seed 仓库维护的 Agent 定义
- 该 Agent 知道自己服务的唯一仓库、主线文档入口和默认迭代流程
- 该 Agent 能把请求归类为：文档重组、能力建设、验收回写、使用说明更新或需人类决策
- 人类完成飞书应用配对后，可直接在飞书中触发该 Agent

## 4. 范围

### 本次范围

- 定义 seed 项目维护 Agent 的职责、边界和工作流
- 定义其最小加载策略和默认输入输出
- 定义其与人类、`OpenClaw`、`ACP/Codex` 的升级条件
- 为后续飞书配对接入准备工件

### 本次不做

- 不直接修改本机 `~/.openclaw` 配置
- 不直接实现飞书应用配对
- 不直接打通自动提交、自动推送或多仓库支持

## 5. 用户场景

1. 作为 seed 仓库维护者
2. 我希望在飞书中直接唤起一个专门维护本仓库的 Agent
3. 以便我不用每次重复解释上下文，也能稳定推进仓库演进

主路径：

- 人类在飞书中请求继续维护 seed 仓库
- `OpenClaw` 识别该请求属于 seed 项目维护
- Agent 加载最少必要文档和当前迭代待办
- Agent 产出下一步建议；凡涉及实现或专业能力，必须通过 `ACP` 调用 `Codex`
- 结果回写到仓库工件，并在需要时回到人类确认

失败路径：

- 若请求超出 seed 仓库范围，Agent 不直接处理
- 若缺少飞书应用配对或权限边界不清，Agent 停在准备状态
- 若变更影响主线协作方式，必须回到人类确认

## 6. 功能性要求

- Agent 必须识别自己的唯一服务对象是 `openclaw-codex-product-os-seed`
- Agent 必须以 `docs/index.md`、`docs/00-story-map/index.md` 和 `docs/current-iteration.md` 作为默认入口
- Agent 必须按需加载，而不是整包注入全部 docs
- Agent 必须能把请求至少分类为：主线规划、OpenClaw 能力建设、Codex 能力建设、治理更新、seed 用法更新
- Agent 必须定义何时只由 `OpenClaw` 处理，何时必须通过 `ACP` 转交 `Codex`
- Agent 必须定义何时需要人类确认
- Agent 必须有独立工作手册和飞书线程工作流

## 7. 非功能性要求

- 性能：默认加载上下文应保持最小化
- 可靠性：分类错误时应回到人类，而不是盲目推进
- 安全或隐私：涉及提交、推送、外部调用时必须遵守升级规则
- 可访问性：飞书中的人类应能看懂当前状态、下一步和阻塞点
- 可观测性：每次迭代应有明确的工件回写位置
