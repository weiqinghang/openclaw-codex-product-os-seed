# 需求简报

## 元数据

- 标题：飞书直连的 seed 项目维护 Agent
- 提出人：人类
- 日期：2026-03-10
- 优先级：高
- 期望时间：当前迭代

## 当前所需上下文

- 当前 seed 仓库的主线文档结构
- 当前 `OpenClaw` 能力建设方向
- 已有 `OpenClaw` 能力包与 backlog
- 人类希望通过飞书直接驱动本项目继续生长

## 当前不需要的上下文

- 与 seed 仓库无关的业务项目
- 尚未接入的设计定版流程
- 远期原生多代理内核改造细节

## 1. 请求概述

定义一个可在飞书中直接调用的 Agent，让人类可以通过 `OpenClaw` 持续维护 `openclaw-codex-product-os-seed`，而不是每次都在仓库内手工组织下一步。

## 2. 用户与场景

- 主要用户：维护 seed 仓库的人类
- 触发场景：人类在飞书里提出“继续完善 seed 项目”的请求
- 当前绕行方式或失败表现：需要先在对话里临时解释上下文，再手动决定改文档、改能力包还是改规格

## 3. 期望结果

人类可以在飞书中直接唤起一个专门维护 seed 项目的 Agent；该 Agent 能识别当前请求、加载最少必要上下文、决定是否调用 `Codex`，并把结果继续沉淀到本仓库。

## 4. 约束

- 时间：先完成最小可用切片，不追求一次打通全部自动化
- 技术：优先基于现有 `OpenClaw` 能力与扩展层实现
- 合规或安全：涉及 Git、飞书、仓库变更时必须保留升级条件
- 依赖：人类后续会创建飞书应用并完成配对

## 5. 证据

- 用户反馈：希望通过 `OpenClaw` 来驱动本项目生长
- 数据信号：当前已具备文档体系与能力包，但缺一个专门的项目维护入口
- 干系人输入：人类已明确该 Agent 的使用场景是 seed 项目维护，而不是通用聊天

## 6. 初始切片建议

先定义这个 Agent 的职责边界、输入输出、加载策略和飞书交互流程，并为后续配对接入准备工件。

## 7. 待确认问题

- 该 Agent 是否独占一个飞书应用 / 账号
- 它是否允许自动提交和推送，还是默认只准备变更
- 它是否只服务 seed 仓库，还是未来扩展到其他产品 OS 仓库
