# Project Constitution

## Purpose

This constitution defines the default product and engineering decision rules in this repository.

## Product Principles

### 1. Outcome over output

Every feature must name the user problem, intended behavior change, and success signal.

### 2. Smallest valuable slice

Prefer the smallest release that creates real value.

### 3. Explicit non-goals

Each spec must say what is intentionally excluded.

## Engineering Principles

### 4. Spec before implementation

User-visible changes require a written spec before coding starts.

### 5. Tests are delivery evidence

Automatically verifiable behavior should be backed by tests.

### 6. Refactor continuously

Leave the code clearer after meaningful changes.

### 7. Keep changes reviewable

Avoid large mixed-purpose diffs.

### 8. Prefer boring architecture

Choose the simplest design that satisfies the need.

## Operating Principles

### 9. Plans must mention risk

Every plan must name technical risk, product risk, and verification strategy.

### 10. Completion requires evidence

A task is complete only when code, tests, and verification notes support it.

### 11. Learning updates the system

Repeated friction should update the templates and process.
