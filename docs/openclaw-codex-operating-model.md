# OpenClaw / Codex 操作模型

## 目标

定义本地 `OpenClaw` 与 `Codex` 的长期分工、职责映射和交互方式，使它们能够稳定承担一支敏捷全功能团队中的不同职责。

这份文档解决的不是某个功能如何实现，而是：

- 人类、`OpenClaw`、`Codex` 三者分别扮演什么角色
- 传统岗位如何映射到这三个固定角色
- `OpenClaw` 与 `Codex` 之间如何发生交互
- 哪些节点必须由人类 review 或拍板

## 固定角色

### 人类

最终负责：

- 产品方向
- 业务取舍
- 优先级
- 验收
- 设计定版
- 发布批准

### OpenClaw

固定扮演前台编排者，负责：

- 接收人类请求
- 收敛问题
- 组织上下文
- 选择当前职责模式
- 判断是否需要人类 review
- 向 `Codex` 下达执行请求
- 把 `Codex` 的结果转译给人类

### Codex

固定扮演后台执行者，负责：

- 技术分析
- 规格落仓
- 代码实现
- 测试与验证
- 证据整理
- 技术方案与实现层决策

`Codex` 可以主导技术决策，但不能替代人类做产品方向、验收和发布决策。

## 职责簇映射

### 1. 方向与价值职责簇

包含：

- 产品负责人
- 业务 owner
- 版本 owner

分工：

- 人类：主责任
- `OpenClaw`：收敛问题、组织方案对比、整理版本规划草案、提示风险
- `Codex`：提供实现成本、依赖和技术影响评估

### 2. 体验与设计职责簇

包含：

- 设计师
- 交互设计
- 信息架构

分工：

- 人类：设计定版主责任
- `OpenClaw`：产出原型草案、交互说明、组织 Figma 提案、准备设计 review
- `Codex`：评估可实现性、给出组件映射、实现约束和前端落地建议

### 3. 技术方案职责簇

包含：

- 架构师
- 技术负责人
- 前后端方案设计

分工：

- `Codex`：主责任
- `OpenClaw`：把问题转成可决策的技术议题，并把结论回传给人类
- 人类：只在技术方案会改变产品边界、周期或风险时拍板

### 4. 质量与验证职责簇

包含：

- QA
- 测试策略
- 验收检查
- 回归判断

分工：

- `Codex`：主责任
- `OpenClaw`：把验证结果翻译成可 review 的结论
- 人类：决定是否通过验收、是否接受已知风险

### 5. 交付与运行职责簇

包含：

- 发布协调
- 回滚判断
- 发布说明

分工：

- 人类：主责任
- `OpenClaw`：整理发布信息、风险摘要和待决策项
- `Codex`：提供发布证据、变更清单、验证结果和回滚建议

## 交互原则

### 1. 人类只主要与 OpenClaw 交互

日常协作中：

- 人类不直接对 `Codex` 下达执行指令
- `OpenClaw` 是唯一协调者
- `Codex` 不与人类长期直接往返

### 2. OpenClaw 负责“解释、翻译、分派、回传”

`OpenClaw` 不只是一个聊天入口，它还负责：

- 判断当前处于项目级还是切片级
- 判断当前需要哪一种职责模式
- 决定是否调用 `Codex`
- 决定是否升级给人类 review
- 把 `Codex` 的原始结果整理成可决策摘要

### 3. Codex 只接收明确任务包

`OpenClaw` 与 `Codex` 的交互必须以任务包为单位，而不是开放式聊天。

每个任务包至少包含：

- 当前阶段
- 当前目标
- 上下文边界
- 预期产出
- 升级条件

## 任务包模型

### 当前阶段

可选阶段应限制在少数固定类别：

- 项目收敛
- 切片定义
- 技术方案
- 实现
- 验证
- 发布准备

### 当前目标

必须是这次唯一要解决的问题，例如：

- 输出首个切片建议
- 补齐当前切片 `spec`
- 完成某批实现任务
- 运行验证并整理风险摘要

### 上下文边界

必须明确允许 `Codex` 使用哪些输入，例如：

- 当前 issue
- 当前切片工件
- 指定代码目录
- Figma 链接
- 相关验证记录

不要把整个项目历史和无关背景一起下发。

### 预期产出

必须是明确结果，而不是泛泛地“看一下”：

- 切片建议
- `brief / spec / plan / tasks` 草案
- 技术方案结论
- 实现补丁
- 测试结果
- 风险摘要
- 发布准备摘要

### 升级条件

任务包必须说明：

- 什么情况下 `Codex` 可以继续自动推进
- 什么情况下必须回到 `OpenClaw`
- 什么情况下必须提交给人类

## Codex 返回结果模型

`Codex` 返回给 `OpenClaw` 的结果必须先被归类，再决定流向。

### 可自动继续

例如：

- 实现完成
- 测试通过
- 只需进入下一步

### 需人类 review

例如：

- 版本规划草案
- 首个切片建议
- 设计定版候选
- 验收结论

### 需人类决策

例如：

- 范围冲突
- 优先级冲突
- 技术方案改变周期
- 风险超阈值

### 需补充输入

例如：

- 缺少设计链接
- 验收标准不完整
- 依赖条件不明

## OpenClaw 的结果转译规则

`OpenClaw` 不应把 `Codex` 的原始执行细节整包转发给人类，而应转译成决策友好的摘要。

摘要至少包含：

- 当前发生了什么
- 为什么需要 review 或决策
- 可选项是什么
- 默认建议是什么
- 不决策会阻塞什么

## 阶段化运行模型

### 1. 项目级阶段

这一阶段由人类主要与 `OpenClaw` 协作。

`OpenClaw` 在这一阶段承担：

- 产品经理职责模式
- 设计协调职责模式
- 版本 owner 职责模式

优先产出：

- 项目目标
- 版本规划草案
- 原型或交互草案
- Figma 提案组织
- GitHub 项目级 issue 草案

必须由人类拍板：

- 项目是否启动
- 首版范围
- 设计定版
- 首个切片是否成立

### 2. 切片级阶段

这一阶段由 `OpenClaw` 把已确认结论压缩成任务包发给 `Codex`。

`Codex` 在这一阶段承担：

- 技术负责人职责模式
- 工程实现职责模式
- QA / 验证职责模式

主要产出：

- `brief / spec / plan / tasks`
- 技术方案
- 实现与测试结果
- 验证证据
- 风险和回滚建议

`OpenClaw` 只在关键节点把结果提交给人类：

- 切片边界确认
- 验收标准确认
- 风险超阈值
- 发布批准

### 3. 例外处理阶段

出现以下情况时，`OpenClaw` 必须中止自动流转并拉人类进来：

- 当前切片无法维持最小边界
- 技术方案改变项目级周期或范围
- 设计定版与实现约束冲突
- 验收失败且需要改动产品边界
- 发布风险超出默认阈值

## 本地 OpenClaw 需要具备的机制

### 1. 前台编排机制

`OpenClaw` 至少需要具备这些能力：

- 阶段识别
- 职责模式切换
- 审批网关控制
- 结果转译
- 工件投递

推荐先用这些文档和模板作为最小实现载体：

- [审批网关模型](./approval-gateway-model.md)
- [OpenClaw 改造 Backlog](./openclaw-backlog.md)
- [OpenClaw 能力包规范](./openclaw-capability-pack-spec.md)
- [Codex 任务包模板](../templates/codex-task-packet.md)
- [人类决策摘要模板](../templates/human-decision-brief.md)

### 2. Codex 调用机制

`OpenClaw` 对 `Codex` 的调用应限制在固定任务类别：

- 技术评估任务
- 切片规格任务
- 实现任务
- 验证任务
- 发布准备任务

每类任务都应有固定输入模板和固定输出模板，避免让 `Codex` 直接承担前台编排职责。

### 3. 审批节点机制

默认只在关键节点拉人类进来：

- 项目启动批准
- 版本边界批准
- 设计定版批准
- 首个切片批准
- 验收通过批准
- 发布批准

其余节点默认由 `OpenClaw` 和 `Codex` 自行流转。

更具体的网关定义见：[审批网关模型](./approval-gateway-model.md)

## 验收场景

- 人类提出新项目想法时，`OpenClaw` 能先以产品与设计协调职责模式收敛问题，而不是直接要求写仓库工件
- 项目方向确认后，`OpenClaw` 能产出项目级 issue 草案，并把它作为后续切片流转的正式入口
- 首个切片确认后，`OpenClaw` 能向 `Codex` 发送一个边界清楚的任务包，而不是把全部项目背景直接下发
- `Codex` 完成技术方案与实现后，结果先回到 `OpenClaw`，由 `OpenClaw` 判断哪些内容继续自动推进，哪些内容提交人类 review
- 当 `Codex` 发现风险会改变项目级范围或周期时，系统会中断自动推进并把决策请求升级给人类
- 设计定版仍以 Figma 为准，`Codex` 只消费设计输入并反馈可实现性，不直接替代人类定版
