# 执行手册

## 目标

把“敏捷 + XP + SDD + 明确角色分工”落成具体执行规则。

## 默认节奏

每次只推进一个切片。

但在进入切片节奏之前，项目需要先完成一次项目级启动。

项目级启动规则见：[项目启动手册](./project-startup-playbook.md)

一个切片的默认节奏是：

1. 先收敛问题
2. 再定义当前验收
3. 再决定当前批次怎么做
4. 做完一批就验证
5. 没过就更新理解，不硬撑到“大完成”

## 切片规则

### 什么叫一个好切片

- 有独立用户价值
- 有独立验收标准
- 能在一次 review 中看懂
- 可以在一轮实现后得到反馈

更具体的检查项见：[切片质量检查清单](./slice-quality-checklist.md)

### 什么情况下要拆切片

出现以下任一情况就应该拆：

- 当前规格包含两个不同用户目标
- 当前计划跨越多个系统边界
- 当前任务超过 3-5 个且相互依赖严重
- 需要先做很多前置工作才能看到结果

## 最后责任时刻

下列内容不要过早决定：

- 远期扩展结构
- 不确定是否会发生的边界场景
- 尚未被当前切片触发的抽象层

下列内容不能拖到太晚：

- 当前切片的验收标准
- 当前切片的验证方式
- 当前切片的风险控制

## 渐进明细规则

### brief 只写这些

- 为什么现在值得做
- 当前主要用户是谁
- 当前最小切片是什么

不要在 `brief` 里提前展开完整方案。

### spec 只写这些

- 当前切片范围
- 当前切片不做什么
- 当前切片验收标准

不要在 `spec` 里写远期 roadmap。

### plan 只写这些

- 当前切片要动哪里
- 当前切片怎么验证
- 当前切片有哪些风险

不要把未来可能的所有实现方案都写进去。

### tasks 只拆当前批次

每个任务都必须：

- 可独立验证
- 可独立 review
- 对当前切片有直接贡献

## 面向 `Codex` 的上下文规则

### 必须给 `Codex` 的内容

- 当前切片的 `brief`
- 当前切片的 `spec`
- 当前切片的 `plan`
- 当前批次的 `tasks`
- 当前代码区域的必要上下文

### 不要默认给 `Codex` 的内容

- 无关 feature 的历史背景
- 未来版本想法
- 尚未进入当前切片的需求
- 与当前改动无关的大篇会议纪要

### 对话规则

- 一个对话优先只解决一个切片
- 一个实现回合优先只完成一批任务
- 如果中途发现理解变了，先更新工件，再继续实现

## 执行步骤

### 第 0 步：确认项目已完成启动

在开始切片执行前，应先确认：

- 人类已在飞书中与 `OpenClaw` 完成项目级收敛
- GitHub 中已存在项目级 issue
- 当前切片已被人类确认值得优先落地

### 第 1 步：开切片 issue

把当前需求或缺陷放进 issue，确认它是否值得成为一个独立切片。

### 第 2 步：创建功能工作区

```bash
make new-feature SLUG=<feature-slug>
```

### 第 3 步：写 brief

只澄清当前问题，不写全量方案。

### 第 4 步：写 spec

只写当前切片，不替未来切片做决定。

### 第 5 步：写 plan

只写当前切片必须知道的设计和风险。

### 第 6 步：写 tasks

只拆当前准备执行的一批任务。

### 第 7 步：执行

执行顺序：

1. 拿一个任务
2. 实现
3. 验证
4. 更新工件
5. 再拿下一个任务

### 第 8 步：校验

```bash
make validate-specs
```

### 第 9 步：评审

评审重点不是“写得像不像原计划”，而是：

- 当前切片是否真的满足验收
- 风险是否在可接受范围
- 是否应进入下一切片

## 完成定义

一个切片只有在以下条件同时成立时才算完成：

- 当前切片代码已完成
- 当前切片验收已验证
- 当前切片风险已说明
- 当前切片工件已更新

切片完成，不等于整个主题完成。
