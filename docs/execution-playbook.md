# 执行手册

## 目标

把仓库里的方法论压缩成一套可以日常执行的操作规则，降低“知道理念，但不会落地”的摩擦。

## 标准执行路径

### 第 1 步：需求进入

1. 先开 GitHub issue
2. 用功能提案或缺陷报告模板收敛问题
3. 判断是否值得进入一个独立交付切片

### 第 2 步：创建功能工作区

```bash
make new-feature SLUG=<feature-slug>
```

规则：

- `slug` 必须是小写 kebab-case
- 一个目录只承载一个交付切片

### 第 3 步：写需求简报

必须回答：

- 这是什么请求
- 谁受影响
- 成功后会发生什么变化
- 当前最小交付切片是什么

### 第 4 步：写功能规格

必须回答：

- 问题是什么
- 哪些场景在本次范围内
- 哪些内容明确不做
- 验收标准是什么

### 第 5 步：写实现计划

必须回答：

- 影响哪些组件
- 如何分批上线
- 如何回滚
- 如何验证

### 第 6 步：写任务拆解

- 每个任务都必须可独立验证
- 每个任务都必须写验证方式
- 优先按用户价值纵切

### 第 7 步：执行

1. 改一个小片段
2. 跑相关检查
3. 更新对应工件
4. 再进入下一片

### 第 8 步：验证

```bash
make validate-specs
```

### 第 9 步：开 PR

PR 中必须写清：

- 改了什么
- 怎么验证
- 还剩什么风险

### 第 10 步：发布与复盘

- 发布前填写发布检查清单
- 发布后记录复盘
- 重复摩擦要更新模板或文档
