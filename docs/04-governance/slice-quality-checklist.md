# 切片质量检查清单

## 目标

用一份简单、可重复使用的检查清单，判断当前 `brief / spec / plan / tasks` 是否仍然服务于“当前切片”，而不是重新滑回“大而全规格”。

这份清单的用途不是卡流程，而是帮助团队更早发现：

- 切片过大
- 决定过早
- 上下文过载
- 任务批次过重

## 什么时候使用

- 写完当前切片工件后，开始实现前
- 实现中发现任务越做越大时
- PR 评审时，需要判断这次变更是不是一个合理切片时
- 准备进入下一切片前，回看本次切片是否拆得合适时

## 快速判断

如果下面 4 个问题里有 2 个以上回答“不确定”或“否”，当前切片通常需要重切：

- 这个切片是否有独立用户价值？
- 这个切片是否有独立验收标准？
- 这个切片是否能在一次 review 中看懂？
- 这个切片是否能在一轮实现后拿到反馈？

## 检查清单

### 1. 价值与边界

- [ ] 当前切片解决的是一个清楚的用户问题，而不是一组松散想法
- [ ] 当前切片有独立用户价值，即使整个主题还没完成也值得交付
- [ ] 当前切片有明确边界，能说清“本次做什么”和“本次不做什么”
- [ ] 当前切片没有同时追两个不同用户目标
- [ ] 当前切片完成后，团队能明确判断是否进入下一切片

出现以下信号时，通常说明切片过大：

- 一个切片里同时想验证价值、补齐架构、铺好未来扩展
- 本次范围跨越多个系统或角色，但没有一个清楚的主路径
- 讨论中频繁出现“顺手也一起做了”

### 2. 验收与反馈

- [ ] 当前切片有可验证的验收标准
- [ ] 验收标准描述的是行为结果，而不是实现过程
- [ ] 做完这一轮后，能够通过测试、演示或真实使用拿到反馈
- [ ] 不需要等后续多个切片全部完成，才能判断这次是否有效
- [ ] 如果失败，团队知道应该更新哪份工件再继续

出现以下信号时，通常说明验收还不够好：

- 验收写成“完成开发”“完成联调”“代码提交”
- 必须等“大版本完成”才能知道当前切片有没有价值
- 没有定义如何验证，只说“应该可以”

### 3. 最后责任时刻

- [ ] 只决定了当前切片必须决定的内容
- [ ] 尚未被当前切片触发的抽象、扩展点、边界场景没有提前定死
- [ ] 当前 `plan` 聚焦在现在要改哪里、怎么验证、有哪些风险
- [ ] 对未来可能发生但尚未触发的问题，只记录假设或待确认问题
- [ ] 没有把“为了以后可能要用”当成当前设计的主要理由

出现以下信号时，通常说明决定过早：

- 在 `spec` 或 `plan` 里详细设计未来 2-3 个切片
- 为尚未出现的变化一次性引入复杂抽象
- 当前用户路径还没验证，就先优化未来扩展结构

### 4. 上下文与 `Codex` 负担

- [ ] 当前工件只包含理解这个切片所需的信息
- [ ] 提供给 `Codex` 的上下文边界清楚，没有混入无关背景
- [ ] 没有把未来 roadmap、无关会议纪要、其他 feature 历史一起塞进来
- [ ] 一个对话主要只处理一个切片或一个批次
- [ ] 当前工件内容足够短，`Codex` 和评审者可以快速定位边界与决策

出现以下信号时，通常说明上下文过载：

- `brief` 大段展开完整方案
- `spec` 同时覆盖当前切片和未来阶段
- `tasks` 混入当前批次之外的“顺便准备项”

### 5. 任务批次与评审成本

- [ ] 当前 `tasks` 只拆当前准备执行的一批任务
- [ ] 每个任务都对当前切片有直接贡献
- [ ] 每个任务都可以独立验证或独立 review
- [ ] 当前批次大小仍然可控，不需要长时间堆积后再统一反馈
- [ ] 评审者可以在一次评审中理解这批任务的目标、风险和结果

出现以下信号时，通常说明批次过大：

- 任务数量明显超过一个合理批次，并且相互依赖严重
- 需要先完成很多前置工作，才能看到第一个可验证结果
- 评审时必须先理解大量未来设计，才能判断当前改动

## 按工件检查

### brief

应该重点检查：

- [ ] 是否只回答“为什么现在值得做”
- [ ] 是否清楚写出当前主要用户、场景和最小切片
- [ ] 是否明确列出当前所需上下文与当前不需要的上下文

需要重写的典型信号：

- `brief` 已经变成完整方案说明书
- `brief` 里塞入多个切片的验收和实现细节

### spec

应该重点检查：

- [ ] 是否清楚写出当前范围与本次不做
- [ ] 是否只包含当前切片的用户场景和验收标准
- [ ] 是否把风险、假设、待确认问题与确定结论区分开

需要重写的典型信号：

- `spec` 试图一次覆盖完整主题 roadmap
- 验收标准无法独立验证

### plan

应该重点检查：

- [ ] 是否只说明当前切片必须知道的设计
- [ ] 是否说明验证方式、风险和影响范围
- [ ] 是否避免为未来切片预支复杂度

需要重写的典型信号：

- `plan` 充满未来阶段的技术铺垫
- 当前实现路径不清楚，但未来扩展设计很详细

### tasks

应该重点检查：

- [ ] 是否只拆当前批次
- [ ] 是否每项任务都可独立验证
- [ ] 是否没有把未来任务提前混进来

需要重写的典型信号：

- 一个任务其实包含多步不可分离工作
- 任务列表像“大里程碑分解”，不是当前批次执行单

## 发现问题后怎么处理

### 情况 1：切片过大

处理方式：

1. 回到 `brief` 和 `spec`
2. 重写“本次范围 / 本次不做”
3. 重新定义最小可验证切片
4. 再更新 `plan` 和 `tasks`

### 情况 2：决定过早

处理方式：

1. 把未来内容改成假设、风险或待确认问题
2. 删除当前切片不必决定的抽象与细节
3. 只保留支撑当前验收所需的设计

### 情况 3：任务批次过重

处理方式：

1. 把当前 `tasks` 拆成更小批次
2. 先保留最近要执行的一批
3. 剩余内容等下一轮反馈后再展开

## 最终判断

满足以下条件时，可以认为当前切片质量基本合格：

- 有独立用户价值
- 有独立验收标准
- 能在一次 review 中看懂
- 能在一轮实现后得到反馈
- 工件只承载当前切片必须知道的内容

如果仍然拿不准，默认继续拆小，而不是继续写大。
