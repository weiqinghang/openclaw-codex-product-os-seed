# OpenClaw 改造 Backlog

## 目标

把针对本地 `OpenClaw` 的长期改造事项集中维护在一个地方，作为本项目的一部分持续迭代。

这份 backlog 关注的是：

- 如何让 `OpenClaw` 更稳定地扮演前台编排者
- 如何让 `OpenClaw` 与 `Codex` 的交互更结构化
- 哪些能力应优先通过配置、扩展或外围层实现
- 哪些能力在扩展点不足时才考虑源码改造

## 使用规则

- 每个条目只描述一种能力改造，不混多个目标
- 优先记录“能力是什么”和“为什么值得做”，而不是先写实现细节
- 默认优先选择“无源码”或“扩展层”方案
- 只有当现有机制明显做不到时，才把条目标记为“可能源码”
- 条目完成后，不删除，改为已完成并补一句结果

## 状态说明

- `todo`：已识别，但尚未开始
- `in-progress`：正在设计或实现
- `done`：已完成并可复用
- `blocked`：已识别，但被依赖或外部条件阻塞

## 当前优先级

当前建议优先推进的条目：

1. `OC-007` 飞书消息路由
2. `OC-010` 上下文裁剪器
3. `OC-011` 结果分类器
4. `OC-013` 审批状态存储
5. `OC-018` 审批网关状态机实现

原因是这几项决定了：

- `OpenClaw` 能否真正稳定扮演前台编排者
- `Codex` 能否只接收边界清楚的执行任务
- 人类能否只在关键节点介入，而不是被执行细节淹没

## 维护说明

- 当新增一项长期能力改造时，先在这里登记，再决定是否进入实现
- 当某项能力的推荐实现层发生变化时，直接更新本表
- 当发现某个“扩展层”能力实际上做不到时，再把它升级为“可能源码”
- 如果后续真的进入本地 `OpenClaw` 仓库实施，应在对应条目后补充关联 issue、PR 或实现位置
- 当某个能力开始产品化时，应同步在 `capabilities/` 下创建对应能力包目录
