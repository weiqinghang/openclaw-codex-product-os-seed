# 综合待办清单

这份清单用于持续迭代本项目本身。

目标不是零散补文档，而是按固定节奏推进：

1. 协作方式规划
2. 能力规划与验收规划
3. 能力建设与术语补齐
4. 验收效果
5. 使用说明更新

---

## 迭代原则

- 每次只推进一个明确主题或一个小批次
- 先收敛协作方式，再进入能力建设
- 每次能力建设都要带验收标准
- 新能力、新术语、新规则进入仓库后必须补齐对应文档
- 验收结论要回到文档，而不是只停留在对话里
- 使用说明只描述当前真实可用的做法

## 人类参与原则

- 人类负责决定当前优先级、范围边界和是否进入下一步
- 人类负责确认验收口径是否成立
- 人类负责决定是否接受已知限制和未完成项
- 需要改动主线协作方式时，必须有人类确认
- 需要把实验能力升级为默认能力时，必须有人类确认

---

## 流程待办

### 1. 协作方式规划

- [ ] 明确当前要优化的是哪一段主线：人类 -> OpenClaw / OpenClaw -> Codex / Codex -> OpenClaw / OpenClaw -> 人类
- [ ] 明确当前问题属于项目启动、切片执行、审批回流还是 seed 使用
- [ ] 明确这次迭代不处理什么，避免一轮里混多个目标
- [ ] 人类确认：当前主题是否值得进入本轮迭代

### 2. 能力规划与验收规划

- [ ] 把当前主题拆成一个或多个能力项
- [ ] 明确每个能力项的目标、边界和推荐实现层
- [ ] 明确每个能力项的验收标准和失败信号
- [ ] 明确哪些节点允许自动推进，哪些节点必须回到人类
- [ ] 人类确认：当前验收标准、风险取舍和默认建议

### 3. 能力建设与术语补齐

- [ ] 实现或更新对应文档、模板、prompt、config、example
- [ ] 若进入产品化，补齐 `capabilities/` 下的能力包资产
- [ ] 若引入新术语，更新 `docs/04-governance/glossary.md`
- [ ] 若引入新规则，更新对应主线文档或治理文档
- [ ] 人类 review：确认能力形态是否符合主线模型

### 4. 验收效果

- [ ] 验证这次改动是否真的让主线更清楚、更可执行
- [ ] 验证是否降低了上下文负担，而不是增加说明复杂度
- [ ] 验证是否引入了新的冲突、重复或死链
- [ ] 记录已知限制、剩余风险和下一轮切入点
- [ ] 人类验收：决定通过、调整后重试或暂缓

### 5. 使用说明更新

- [ ] 更新仓库根 `README.md` 的入口说明
- [ ] 更新相关目录的 `index.md`
- [ ] 更新下游使用方式，只保留当前真实有效的操作
- [ ] 清理已经过期的表述、路径和示例
- [ ] 人类确认：外部读者是否能用最短路径理解并上手

---

## 当前执行规则

- 当某一轮迭代开始时，先在本清单中圈定本轮关注项
- 当某一轮能力建设完成时，必须回写验收结果
- 当某一轮发现流程本身需要调整时，优先更新本清单，再更新其他文档
