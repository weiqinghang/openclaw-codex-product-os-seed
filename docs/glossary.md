# 术语表

## 使用规则

- 本表用于统一仓库中的产品、研发、流程术语含义。
- 当文档、模板、示例、issue 模板、PR 模板中出现新的专有术语时，必须同步补充本表。
- 路径、文件名、命令、变量名仍然保持英文，不在本表中重复解释基础编程语法词汇。

## 术语

### OpenClaw

需求入口、协作入口和编排入口。负责接收原始产品意图，并把问题整理成适合进入仓库流程的请求。

### Codex

仓库内执行代理。负责阅读代码和工件、制定实现方案、修改代码、运行检查并产出可审查结果。

### XP

极限编程。这里主要指小步快跑、快速反馈、测试优先、持续重构。

### SDD

规格驱动开发。核心思想是先澄清需求和行为，再进入实现。

### Spec-Kit

偏向新能力或大功能启动时使用的规格化方法，强调从原则、规格、计划到任务的串联。

### OpenSpec

偏向已有产品持续演进时使用的变更表达方式，强调需求、设计和实施任务之间的可审查关系。

### Superpowers

这里指一组强化执行纪律的工作习惯，例如 brainstorm、write-plan、execute-plan、verify。

### brief

需求简报。用于把原始请求收敛成“问题、用户、目标、约束、初始切片”的最小表达。

### spec

功能规格。用于定义问题、范围、用户场景、功能要求、非功能要求和验收标准。

### plan

实现计划。用于说明设计方案、影响范围、风险、交付策略和测试策略。

### tasks

任务拆解。用于把计划拆成一组可独立验证的小任务。

### retro

复盘。用于记录本次迭代的结果、摩擦、风险和流程改进点。

### iteration

迭代。指一次小范围推进和反馈循环，不追求一次性解决整个主题。

### CTA

`Call To Action` 的缩写，意思是引导用户执行下一步动作的按钮、链接或入口。

在产品界面里，常见形式包括：

- “立即创建”
- “开始配置”
- “查看指南”

### slice

切片。指一个足够小、但又真实有价值的交付单元。

### delivery slice

交付切片。指这次迭代准备独立完成的一小块可交付价值，通常对应一个 feature 工作目录。

### progressive elaboration

渐进明细。指越接近实现的内容越具体，而不是在一开始就把所有细节写满。

### last responsible moment

最后责任时刻。指把高成本决定尽量延后到必须决定的时刻，但不拖到影响当前切片交付。

### context boundary

上下文边界。指当前切片或任务真正需要依赖的信息范围，用来避免向 AI 提供无关背景。

### feature workspace

功能工作区。指 `specs/features/<feature-slug>/` 这一组工件目录。

### slug

用于命名 feature 目录或其他标识的短名称，要求使用小写 kebab-case，例如 `improve-onboarding`。

### kebab-case

一种命名方式，单词全部小写，并用连字符连接，例如 `openclaw-product-os`。

### scope

范围。指当前这次交付明确要做的部分。

### out of scope

本次不做。指当前规格里明确排除、不在本次交付内的内容。

### acceptance criteria

验收标准。用于判断功能是否真正满足需求的可验证条件。

### rollout

发布方式。指功能如何逐步生效，例如全量发布、灰度发布、开关控制。

### rollback

回滚方式。指当发布后出现问题时，如何快速撤回或关闭该功能。

### feature flag

功能开关。用于在不回滚整套代码的前提下，控制某个功能是否生效。

### observability

可观测性。指日志、指标、事件等帮助团队理解系统状态和行为的能力。

### dashboard

仪表盘或主工作台页面。通常指用户进入产品后最核心的概览页面。

### workspace

工作区。指用户在产品中进行协作、组织内容或执行任务的基本空间。

### issue

GitHub issue。用于记录需求、问题、缺陷或流程改进入口。

### PR

`Pull Request` 的缩写。用于提交变更、链接相关工件，并进行评审与合并。
