# 项目启动手册

## 目标

定义一个新项目从“人类在飞书中提出想法”到“`Codex` 在仓库中开始推进首个切片”的标准启动方式。

这份手册要解决的不是某个切片怎么实现，而是：

- 项目级问题先在哪里讨论
- 哪个系统是正式入口
- 哪些内容由 `OpenClaw` 先产出
- 哪个节点之后 `Codex` 才能开始落仓写工件

## 最终形态

项目启动后的职责边界固定如下：

- 人类：决定方向、优先级、边界、验收和发布
- `OpenClaw`：在飞书中收敛问题、组织上下文、产出前期草案、提议切片
- `Codex`：在 GitHub 入口确认后，落仓写入并维护切片工件，推进实现与验证

系统边界固定如下：

- 飞书：人类与 `OpenClaw` 的主协作界面
- GitHub：正式项目入口和正式工件入口
- Figma：设计定版来源
- 仓库中的 `brief / spec / plan / tasks`：切片级执行载体

## 项目启动流程

### 第 1 阶段：飞书中收敛项目方向

人类在飞书中向 `OpenClaw` 发起项目想法、业务目标或问题。

这一阶段由 `OpenClaw` 先产出草案供人类 review，内容通常包括：

- 项目目标与成功标准
- 目标用户与核心场景
- 首版范围与明确不做
- 版本规划草案
- 原型草案、流程草图、交互说明
- 需要时的 Figma 高保真设计提案

这一阶段的目标不是写仓库工件，而是把“项目是否成立、先做什么、先不做什么”收敛清楚。

### 第 2 阶段：建立 GitHub 项目级 issue

当人类认可方向后，由 `OpenClaw` 把结论沉淀为一个 GitHub 项目级 issue，作为正式启动载体。

该 issue 至少应包含：

- 项目目标
- 目标用户与核心场景
- 首版范围与明确不做
- 关键风险、依赖、待确认问题
- 版本规划摘要
- 飞书文档链接
- Figma 链接
- 首个切片候选

规则是：

- 飞书保留讨论过程
- GitHub 项目级 issue 作为正式记录
- 没有项目级 issue 时，不进入仓库切片工件阶段

### 第 3 阶段：确认首个切片

项目级 issue 建立后，由 `OpenClaw` 基于该 issue 提出首个最小交付切片建议。

人类需要确认：

- 这个切片是否值得优先落地
- 切片边界是否足够小
- 当前验收是否清楚
- 是否仍有项目级问题没有澄清

只有在该切片被人类确认后，`Codex` 才开始进入仓库落地工件。

### 第 4 阶段：`Codex` 落仓写切片工件

切片确认后，由 `Codex` 创建或更新当前切片的：

- `brief`
- `spec`
- `plan`
- `tasks`

此时的职责边界是：

- `Codex` 负责写入和维护文件
- 人类负责最终确认边界、验收和风险取舍
- `OpenClaw` 继续补充上下文和推动收敛，而不是直接替代仓库工件

### 第 5 阶段：进入持续迭代

项目启动完成后，同时存在两条节奏：

- 项目级节奏：人类与 `OpenClaw` 在飞书中维护版本规划、设计方向、优先级变化和下一个切片候选
- 切片级节奏：`Codex` 在仓库中维护当前切片工件、当前批次实现、验证证据和完成状态

## 使用规则

### 飞书里做什么

- 提出项目想法
- 澄清业务目标
- 讨论版本规划
- review 原型草案和高保真设计
- 调整优先级和方向

不要在飞书里把聊天记录直接当作正式交付工件。

### GitHub 里做什么

- 建立项目级 issue
- 记录正式项目目标、范围和风险
- 关联飞书文档和 Figma
- 记录首个切片候选和后续切片入口

GitHub 是正式项目入口，不是飞书讨论的镜像备份。

### Figma 里做什么

- 保存页面结构和交互
- 保存视觉设计和高保真稿
- 作为设计定版来源

仓库中只保留相关链接和切片说明，不维护一套平行设计文档。

### 仓库里做什么

- 只承接已确认的切片
- 只写当前切片所需的 `brief / spec / plan / tasks`
- 只维护当前切片的实现、验证和证据

不要把整个项目的模糊讨论直接写进切片工件。

## 验收场景

- 人类在飞书里提出一个新项目想法时，`OpenClaw` 能先产出项目草案供 review，而不是直接要求人类手写仓库规格
- 项目方向确认后，存在一个单独的 GitHub 项目级 issue 作为正式启动入口
- 项目级 issue 存在时，`OpenClaw` 能提出首个交付切片建议，而不是直接把整个项目落成一套大规格
- 首个切片被确认后，`Codex` 才开始在仓库中写入 `brief / spec / plan / tasks`
- Figma 被作为设计定版来源引用，仓库中只保留相关链接和必要说明
- 项目级讨论继续留在飞书与项目级 issue，切片级执行继续留在仓库工件，职责不混淆
