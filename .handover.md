## 项目初衷

这个仓库的目标不是单纯沉淀一套“AI 写代码模板”，而是把 `OpenClaw + Codex` 组织成一套围绕产品研发全流程的种子项目。

目标包括：

- 用敏捷价值观约束交付方式，而不是滑向瀑布式规格流
- 用 XP 的小步快跑、快速反馈、测试优先、持续重构思想约束执行
- 用 SDD 管理交付证据，让 `brief / spec / plan / tasks / validation` 成为真实工作载体
- 让这套流程更适合 AI 使用：
  - 上下文最小化
  - 工件职责清晰
  - 每次对话和每次实现尽量只处理一个切片或一个批次
- 最终把它作为 GitHub seed repository，供后续项目反复引用和演化

## 当前关键判断

以下判断已经基本确定，后续 session 应默认沿此方向继续：

### 1. 不要回到双语并行维护

当前阶段不要重新维护中英文双版本文档。先把中文版本打磨稳定。

### 2. 不要回到瀑布式大规格

不要把流程重新写成：

- 先完整分析
- 再完整设计
- 再完整拆任务
- 最后统一实现

这会背离当前目标。

### 3. 规格要服务于当前切片

`brief / spec / plan / tasks` 都应该只服务于当前切片，而不是试图覆盖完整主题。

### 4. AI 友好是硬约束

任何后续文档优化，都应该考虑：

- 是否把职责切清楚
- 是否减少不必要上下文
- 是否便于 AI 在单回合内工作
- 是否避免把未来内容提前塞进当前切片

## 下一步可以做什么

以下是当前最值得继续推进的方向，按优先级排序：

### 优先级 1：增加“切片质量检查清单”

这是当前最值得做的下一步。

目标：

- 判断一个 `brief/spec/plan/tasks` 是否过大、过早、过度设计
- 防止这套流程在真实使用中再次滑回瀑布

建议落点：

- 新增 `docs/slice-quality-checklist.md`
- 或在 `templates/` 中增加对应检查模板

建议内容：

- 这个切片是否有独立用户价值
- 是否能在一次 review 中看懂
- 是否把未来切片内容提前写进来了
- 是否存在可以延后到最后责任时刻再决定的内容
- 当前任务是否已经超过一个合理批次

### 优先级 2：增加“AI 任务输入规范”

目标：

- 规定每次交给 AI 的最小输入包
- 限制默认输入内容，避免上下文过载

建议落点：

- 新增 `docs/ai-task-input-spec.md`

建议内容：

- 当前切片必须提供哪些工件
- 哪些背景信息默认不要提供
- 什么时候必须先更新工件再继续实现
- 一个对话的推荐处理粒度

### 优先级 3：增加“PR 评审时的 XP / SDD 检查项”

目标：

- 让 review 不只看代码
- 同时看切片大小、验收标准、验证证据、上下文边界

建议落点：

- 更新 `.github/PULL_REQUEST_TEMPLATE.md`
- 或新增单独的 review checklist 文档

### 优先级 4：让 GitHub Actions 检查更贴近当前方法论

当前校验主要是结构校验。

后续可以增强为：

- 检查是否存在明显过大切片
- 检查模板中的 AI 边界字段是否填写
- 检查 spec / plan 是否缺关键切片信息

但这一步应该晚于切片质量规则本身的确定。
