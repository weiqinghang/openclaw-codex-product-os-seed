## 项目初衷

这个仓库的目标不是单纯沉淀一套“AI 写代码模板”，而是把 `OpenClaw + Codex` 组织成一套围绕产品研发全流程的种子项目。

目标包括：

- 用敏捷价值观约束交付方式，而不是滑向瀑布式规格流
- 用 XP 的小步快跑、快速反馈、测试优先、持续重构思想约束执行
- 用 SDD 管理交付证据，让 `brief / spec / plan / tasks / validation` 成为真实工作载体
- 让这套流程更适合 `OpenClaw + Codex` 使用：
  - 上下文最小化
  - 工件职责清晰
  - 每次对话和每次实现尽量只处理一个切片或一个批次
- 最终把它作为 GitHub seed repository，供后续项目反复引用和演化

## 当前关键判断

以下判断已经基本确定，后续 session 应默认沿此方向继续：

### 1. 不要回到双语并行维护

当前阶段不要重新维护中英文双版本文档。先把中文版本打磨稳定。

### 2. 不要回到瀑布式大规格

不要把流程重新写成：

- 先完整分析
- 再完整设计
- 再完整拆任务
- 最后统一实现

这会背离当前目标。

### 3. 规格要服务于当前切片

`brief / spec / plan / tasks` 都应该只服务于当前切片，而不是试图覆盖完整主题。

### 4. 面向 `OpenClaw + Codex` 的上下文约束是硬约束

任何后续文档优化，都应该考虑：

- 是否把职责切清楚
- 是否减少不必要上下文
- 是否便于 `OpenClaw` 编排和 `Codex` 执行
- 是否避免把未来内容提前塞进当前切片

### 5. 优先做无源码可复用能力，不要先绑定本地私有实现

关于 `OpenClaw` 的后续改造，当前方向已经明确：

- 优先把能力做成可复制、可安装、可启用、可停用、可升级的能力包
- 优先选择配置、模板、prompt、脚本和外围适配层
- 不要一开始就把方法论绑死在某个本地 `OpenClaw` 私有 fork 上

### 6. backlog 应逐步映射为 capability pack

`docs/openclaw-backlog.md` 不应长期只是需求清单。

后续应逐步把：

- backlog 条目
- 对应能力包 ID
- `capabilities/<capability-slug>/`

串起来，让这个 seed 仓库真的具备下游复用价值。

### 7. OpenClaw 与 Codex 的建设要并行保留，不要只推进一条线

当前已经明确：

- `OpenClaw` 这条线负责前台编排、审批、路由和能力包产品化
- `Codex` 这条线负责方法、设计、验证、协作等执行能力基线

后续 session 不能只盯着其中一条线，而把另一条线遗忘。

### 8. Codex 的能力建设应先走“基线 + backlog + 发现流程”

当前阶段不要直接假定某个 skill 已经可用。

后续应优先按这条顺序推进：

1. 在 `docs/codex-backlog.md` 登记
2. 在 `docs/codex-capability-baseline.md` 判断优先级
3. 在 `docs/codex-skill-sourcing-playbook.md` 记录来源、接入和验证方式

### 9. 现阶段先不要急着为 Codex 建 capability pack 体系

和 `OpenClaw` 不同，`Codex` 当前更需要先补“技能发现、安装、接入、验证”流程。

也就是说：

- `OpenClaw` 当前优先解决 capability pack 复用问题
- `Codex` 当前优先解决 skill / MCP / 工具基线问题

等 `Codex` 的基线足够稳定后，再考虑是否产品化成自己的可复用能力包体系。

### 10. Codex 的技能发现来源已经明确，不要重新发散找入口

当前 session 已经明确：

- `ComposioHQ/awesome-claude-skills`
- `travisvn/awesome-claude-skills`
- `nextlevelbuilder/ui-ux-pro-max-skill`

后续 session 应先围绕这些来源推进，而不是重新从零开始找 skills 阵地。
